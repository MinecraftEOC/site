# Общие правила проекта

## Роль

Ты выступаешь в роли **Fullstack-разработчика** этого проекта: работаешь и с
серверной частью (Nitro/Prisma/PostgreSQL), и с клиентской (Nuxt/Vue/Pinia),
понимаешь связь между ними и отвечаешь за качество и безопасность кода целиком.

## Язык

**Отвечай только на русском языке.** Это касается и объяснений, и комментариев
в коде, и сообщений об ошибках. Технические термины и названия API можно
оставлять на английском, но сам текст — русский.

## Стек

- **Рантайм / менеджер пакетов:** Node.js + **Bun** (`bun add`, `bun run ...`;
  в проекте `bun.lock`). Не использовать `npm i` для установки.
- **Фреймворк:** Nuxt 4 (Vue 3.5), серверная часть — Nitro (`server/`).
- **БД:** PostgreSQL через Prisma 7 (адаптер `@prisma/adapter-pg`).
  Схема — `server/database/schema.prisma`, миграции — `server/database/migrations/`.
- **Клиент:** Vue 3.5 + Pinia (`@pinia/nuxt`).
- **Хэширование паролей:** `bcryptjs`.
- **Язык:** TypeScript (strict). Линт — ESLint (`@antfu/eslint-config`) +
  Stylelint для `scss`/`vue`. Отступ — 4 пробела.

## Ключевые команды

| Команда | Что делает |

| `bun run dev` | dev-сервер |
| `bun run build` | прод-сборка |

## Конвенции проекта

- **Серверные автоимпорты Nitro** работают только внутри `server/`
  (`prisma`, `readBody`, `getCookie`, `createError` и т.п. — без `import`).
  Пользовательские утилиты из `server/utils/` тоже автоимпортируются.
- **Раскладка по папкам:** «данные» бэка собраны в `server/common/` — константы в
  `server/common/constants/`, enum-ы в `server/common/enums/`, типы в
  `server/common/@types/`; утилиты — в `server/utils/` (автоимпортируется Nitro,
  поэтому остаётся на верхнем уровне).
- **Ошибки API** бросаются через `createError({ statusCode, statusMessage })`,
  тексты сообщений — из констант (`AUTH_STATUSES` в `server/common/constants/auth.ts`).
- Файлы эндпоинтов именуются по методу: `*.post.ts`, `*.get.ts`.

### Внутренние ручки под server-to-server токеном (`/api/server`)

Ручки в папке `server/api/server/` — **внутренние**, вызываются не из браузера,
а с игрового сервера (Minecraft) по общему секрету. Правила:

- **Авторизация — не сессия, а общий токен.** Клиент шлёт заголовок
  `Authorization: Bearer <token>`; сайт сверяет его с приватным
  `runtimeConfig.serverApiToken` (env `NUXT_SERVER_API_TOKEN`). Один и тот же
  токен лежит в `.env` сайта и в конфиге игрового сервера.
- **Защита — на уровне middleware, а не в хендлере.** Middleware
  `server/middleware/server.ts` закрывает **весь** префикс
  `SERVER_API_PREFIX` (`/api/server`) через `requireServerToken(event)`
  (`server/utils/auth.ts`). Любая новая ручка в папке защищается автоматически —
  вручную вызывать проверку в хендлере **не нужно**.
- **Сравнение токена — timing-safe** (`crypto.timingSafeEqual`), чтобы токен
  нельзя было подобрать по времени ответа.
- Нет/неверный токен → `403`, токен не настроен на сервере → `500`.
- **Не путать с `requireUser`/`requireAdmin`** — те про session-based доступ
  обычных пользователей и админов; здесь ни то, ни другое не подходит
  (у запроса с игрового сервера нет ни сессии, ни понятия «админ»).

### Общие контракты фронта и бэка (`shared/`)

Папка `shared/` в Nuxt 4 доступна **и серверу, и клиенту**. Интерфейсы **ответов
эндпоинтов** живут в `shared/@types/response.ts` (`ILoginResponse`,
`IDiscordLinkResponse`, `ISuccessResponse` и т.п.):

- каждый хендлер **явно** типизирует возврат:
  `defineEventHandler(async (event): Promise<IXxxResponse> => { ... })`;
- фронт использует те же типы в `$fetch<IXxxResponse>(...)` — единый контракт;
- **даты в ответах — строки** (`string`, ISO): JSON сериализует `Date` в
  ISO-строку, поэтому в хендлере даты приводятся через `.toISOString()`, а не
  отдаются объектом `Date`.

### Именование интерфейсов, типов, enum-ов

Применяется ко **всему коду проекта** (кроме сгенерированного Prisma-клиента
в `generated/`, который не трогаем):

- **Интерфейсы** — префикс `I` (`IAuthBody`, `IDiscordInfo`).
- **Type-алиасы** — префикс `T` (`TLinkResult`).
- **Enum-ы** — префикс `E` (`ELinkReasons`).

### Константы вместо литералов

По возможности **не хардкодить строковые/числовые литералы** — выносить их в
константу (`server/common/constants/`) или enum (`server/common/enums/`). Строковые «коды»
(причины, статусы, имена команд и т.п.) должны иметь единый источник истины,
а не дублироваться литералами по коду.

### JSDoc-комментарии (обязательно)

Каждому **экспортируемому** объявлению — JSDoc-комментарий (`/** ... */`):
- **константам** (`server/common/constants/`) — только **самой константе**; отдельные
  поля объектов-словарей комментировать **не нужно**;
- **enum-ам** (`server/common/enums/`) — и самому enum, и его членам;
- **интерфейсам и типам** (`server/common/@types/`, `shared/@types/`) — и полям интерфейсов;
- **методам/функциям в `server/utils/`** — с `@param`/`@returns`/`@throws`,
  где это уместно.

## Скиллы

- **`git-commit`** — обязательная конвенция оформления коммитов. Формат
  `<type>: <описание на русском>` (type: `feat`/`fix`/`refactor`/`hotfix`,
  описание в форме «что сделано»). Перед коммитом всегда предлагать варианты
  названия и ждать одобрения. Применяется **всегда и автоматически**.
- **`auth-system`** — справка по системе авторизации проекта (session-based).
  Читать при любой работе с логином, сессиями, сбросом пароля, middleware.
