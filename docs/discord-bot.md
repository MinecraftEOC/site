## Что это и зачем

Пользователь сайта может **привязать** свой аккаунт к Discord. Схема:

1. На сайте пользователь запрашивает одноразовый **код**.
2. Идёт в определённый **канал** Discord-сервера (не в личку) и вызывает
   slash-команду `/verify <код>`.
3. Если код верный и не протух — аккаунты связываются: в БД сохраняются данные
   Discord-пользователя, статус привязки становится `LINKED`.

Ввод кода реализован через **slash-команду**, а не обычное сообщение — это не
требует привилегированного интента `MESSAGE CONTENT` и позволяет отвечать
**эфемерно** (ответ видит только вызвавший, код не светится в общем чате).

## Архитектура

Бот живёт **внутри Nitro** как плагин (`server/plugins/discord-bot.ts`), а не
отдельным сервисом. Что это даёт:

- **Один процесс, один деплой** — бот стартует вместе с сервером сайта.
- **Прямой доступ к `prisma`** — бот работает с БД напрямую, без HTTP-моста между
  ботом и бэкендом.

Бот подключается к Discord **исходящим постоянным WebSocket-соединением
(Gateway)**. Discord сам присылает по нему события (в т.ч. вызовы slash-команд),
поэтому **публичный URL / туннель не нужны** — всё работает даже с локалхоста.

> ⚠️ **Хостинг:** такой встроенный бот требует long-running Node-сервера
> (Nitro preset `node-server`, self-host / VPS). На serverless (Vercel/Netlify)
> постоянное Gateway-соединение работать не будет.

## Как работает привязка (поток)

```
Сайт                          Бэкенд (Nitro)                Discord
 │                                  │                          │
 │  «Привязать Discord»             │                          │
 ├─── POST /api/discord/link ──────►│                          │
 │                                  │ generateVerifyCode()     │
 │                                  │ upsert DiscordAccount    │
 │◄──── { code, expiresAt } ────────┤ (status=PENDING)         │
 │                                  │                          │
 │  показываем код + таймер         │                          │
 │  и поллим статус каждые 5–10 с   │                          │
 │                                  │                          │
 │            пользователь пишет /verify <код> в канале ───────►│
 │                                  │◄─── InteractionCreate ────┤
 │                                  │ linkDiscordByCode()      │
 │                                  │ проверка + транзакция    │
 │                                  │ (status=LINKED)          │
 │                                  ├──── эфемерный ответ ─────►│
 │                                  │                          │
 │  поллинг GET /api/discord/status → status === 'LINKED'      │
 │  → останавливаем поллинг, обновляем UI                      │
```

## API эндпоинты сайта

Оба эндпоинта защищены `requireUser` (нужна активная сессия). Типы ответов лежат
в `shared/types/response.ts` и используются как на сервере (явный тип возврата
хендлера), так и на клиенте (`$fetch<IXxx>(...)`).

### `POST /api/discord/link` — запросить код

- Если аккаунт уже `LINKED` → `409` (`DISCORD_STATUSES.ALREADY_LINKED`).
- Иначе генерирует новый `verifyCode` + `verifyExpiry` (now + TTL) и **upsert**-ит
  строку `DiscordAccount` по `userId` (status остаётся `PENDING`).
- Повторный вызов **перезаписывает** старый код новым (старый сразу инвалидируется).
- Ответ — `IDiscordLinkResponse`: `{ code, expiresAt }` (дата — ISO-строка).

### `GET /api/discord/status` — узнать статус (для поллинга)

- Возвращает текущий статус привязки как **данные**, всегда `200`:
  `IDiscordStatusResponse` — `{ status }` (`PENDING` / `LINKED`).
- Это **статус-эндпоинт**: он не бросает ошибку на `LINKED` — наоборот, именно
  этого значения ждёт фронт, опрашивая эндпоинт.

### Поллинг на фронте

После показа кода фронт опрашивает `/status` каждые 5–10 секунд:

```ts
const { status } = await $fetch<IDiscordStatusResponse>('/api/discord/status');
if (status === 'LINKED') {
    stopPolling(); // привязка прошла — обновляем UI
}
```

Условия остановки поллинга:

- `status === 'LINKED'` — успех;
- истёк `expiresAt` из ответа `/link` — код протух, показать «запросите код заново»;
- размонтирование компонента — обязательно очищать интервал (и желательно ставить
  на паузу при `document.hidden`, чтобы фоновая вкладка не слала запросы).

## Настройка в Discord Developer Portal

Разово, вручную. В итоге получаешь 4 значения для `.env` (см. следующий раздел).

### 1. Создать приложение

1. Открой **https://discord.com/developers/applications**.
2. Справа вверху — **New Application**, задай имя, прими соглашение → **Create**.

### 2. Application ID → `NUXT_DISCORD_CLIENT_ID`

1. Вкладка **General Information** → поле **Application ID** → **Copy**.

### 3. Бот и токен → `NUXT_DISCORD_BOT_TOKEN`

1. Вкладка **Bot**. Если бот не создан автоматически — **Add Bot**.
2. Раздел **Token** → **Reset Token** → подтвердить → **Copy**.
   Токен показывается **один раз** — сразу копируй.
3. **MESSAGE CONTENT INTENT включать НЕ нужно** — используем slash-команды.

> 🔒 Токен — это пароль бота. Не коммить в git (у нас он только в `.env`).
> Потерял — снова **Reset Token**, старый перестанет работать.

### 4. Ссылка-приглашение

1. Вкладка **OAuth2 → URL Generator**.
2. **Scopes:** ☑ `bot`, ☑ `applications.commands`.
3. **Bot Permissions:** ☑ `View Channels`, ☑ `Send Messages`.
4. Внизу **Generated URL** → **Copy**.

### 5. Пригласить бота на сервер

1. Открой скопированную ссылку в браузере.
2. В окне **Add to Server** выбери свой сервер → **Continue** → **Authorize**.

> Достаточно права **«Управление сервером» (Manage Server)** — оно входит в права
> администратора. Быть владельцем сервера не обязательно. Если приложение с
> выключенным **Public Bot** — ничего страшного, ты как владелец бота всё равно
> его добавишь.

### 6. Включить Developer Mode

Нужно, чтобы копировать ID. Discord → **Настройки пользователя → Advanced →
Developer Mode** (включить).

### 7. ID сервера → `NUXT_DISCORD_GUILD_ID`

Правый клик по иконке сервера → **Copy Server ID**.

### 8. ID канала → `NUXT_DISCORD_CHANNEL_ID`

Правый клик по текстовому каналу, где будут вводить `/verify` → **Copy Channel ID**.

## Переменные окружения

Секреты в код не пишем — они живут в `.env` (файл в `.gitignore`).

```dotenv
NUXT_DISCORD_BOT_TOKEN=токен_из_шага_3
NUXT_DISCORD_CLIENT_ID=application_id_из_шага_2
NUXT_DISCORD_GUILD_ID=id_сервера_из_шага_7
NUXT_DISCORD_CHANNEL_ID=id_канала_из_шага_8
```

В `nuxt.config.ts` ключи объявлены пустыми строками — это **плейсхолдеры**. Nuxt
подставляет реальные значения из env по правилу **`NUXT_` + КЛЮЧ_В_СНЕЙК_КЕЙСЕ**:

| Ключ в `runtimeConfig` | Env-переменная |
|---|---|
| `discordBotToken` | `NUXT_DISCORD_BOT_TOKEN` |
| `discordClientId` | `NUXT_DISCORD_CLIENT_ID` |
| `discordGuildId` | `NUXT_DISCORD_GUILD_ID` |
| `discordChannelId` | `NUXT_DISCORD_CHANNEL_ID` |

В коде значения читаются через `useRuntimeConfig()`.

## Как устроен код бота

Файл `server/plugins/discord-bot.ts` — Nitro-плагин, выполняется один раз при
старте сервера.

1. **Проверка env.** Если хоть одного из 4 значений нет — плагин пишет
   предупреждение и выходит (`return`). Сайт при этом работает, бот — нет.
2. **Singleton через `globalThis`.** Ссылка на клиент хранится в глобальном
   объекте, чтобы HMR в dev не создавал повторные подключения к Gateway (тот же
   приём, что и для `prisma`).
3. **Создание клиента** с интентом `Guilds` — минимум, нужный для приёма
   slash-команд. Привилегированные интенты не используются.
4. **Описание команды** через `SlashCommandBuilder`: имя `verify`, обязательный
   строковый аргумент `code`. Это только *описание*, не обработчик.
5. **Регистрация команды** на событии `ClientReady` через
   `rest.put(Routes.applicationGuildCommands(...))`. Команды **guild-уровня**
   применяются мгновенно (глобальные Discord раскатывает до часа).
6. **Обработка `InteractionCreate`:**
   - пропускаем всё, кроме команды `verify`;
   - если канал не совпадает с `NUXT_DISCORD_CHANNEL_ID` → эфемерный отказ;
   - иначе достаём `code`, вызываем `linkDiscordByCode(code, { id, username, avatar })`;
   - отвечаем эфемерно: успех либо текст причины по `DISCORD_STATUSES[result.reason]`;
   - любые неожиданные ошибки логируются, пользователю — нейтральный текст.
7. **`client.login(token)`** открывает Gateway-соединение. После успешного логина
   срабатывает `ClientReady` (регистрация команды) и начинают приходить
   взаимодействия. При ошибке логина ссылка в `globalThis` сбрасывается.

Сама привязка — в `server/utils/discord.ts`, функция `linkDiscordByCode`:
ищет `DiscordAccount` по коду со статусом `PENDING` и не протухшим сроком,
проверяет, что Discord не занят другим аккаунтом, и в одной транзакции
(`prisma.$transaction`) проставляет `LINKED` + данные Discord, гася код.
Функция **автоимпортируется** Nitro, поэтому в плагине вызывается без `import`.

## Внесение изменений: что где менять

Настройки бота живут в **трёх местах** (код / Developer Portal / сервер Discord),
и не всё «подтягивается само».

| Что меняешь | Код | Сервер Discord | Портал |
|---|:---:|:---:|:---:|
| Имя / аватар / описание бота | — | — | ✔ (подтянется само) |
| Токен (сброс) | `.env` | — | ✔ |
| Интенты | ✔ `intents: [...]` | — | ✔ (оба, синхронно) |
| Права / scopes | — | переприглашение или роль | ✔ (новая ссылка) |
| Команда `/verify` и её аргументы | ✔ | — | — |

Ключевые правила:

- **Косметика** (имя/аватар/описание) — только портал, подтянется сама.
- **Интенты** — включать и в портале, **и** в коде (`new Client({ intents: [...] })`),
  значения должны совпадать, иначе бот не залогинится или не получит события.
- **Права/scopes** — это не код: сгенерируй новую ссылку-приглашение и пройди по
  ней заново (бот не задублируется), либо выдай право его роли в настройках сервера.
- **Slash-команда** (имя, аргументы, описание) определяется **только в коде** — в
  портале раздела для команд нет. Достаточно поправить `SlashCommandBuilder` и
  **перезапустить** сервер: на старте команда перерегистрируется через `rest.put`.

## Локальное тестирование

Публичный URL не нужен — бот сам «дозванивается» до Discord.

1. Заполни `.env` (4 значения выше). Локальный Postgres должен быть поднят
   (`DATABASE_URL`).
2. `bun run dev`.
3. В консоли появится строка вида
   `[discord-bot] Залогинен как ..., команда /verify зарегистрирована`.
   В выбранном канале при вводе `/` появится команда **/verify**.
4. Проверить бота можно **не дожидаясь фронта**: в `bun run db:studio` создай
   строку `DiscordAccount` со статусом `PENDING`, известным `verifyCode` и
   `verifyExpiry` в будущем. Затем вызови `/verify <код>` — статус должен стать
   `LINKED`.
5. Негативные кейсы: неверный/просроченный код → «Код неверный или устарел»;
   `/verify` в другом канале → «Команда доступна только в канале верификации»;
   тот же Discord на второй аккаунт → «Этот Discord уже привязан…».

> ⚠️ Не держи локального и продового бота с **одним токеном** одновременно — будут
> две Gateway-сессии, и на команду ответят обе. Для локалки заведи отдельный
> тестовый бот/токен (желательно и отдельный тестовый сервер).

## Диагностика проблем

| Симптом | Вероятная причина |
|---|---|
| `[discord-bot] Пропущены env-переменные Discord — бот не запущен` | Не создан `.env`, опечатка в имени переменной или dev-сервер не перезапущен после правки `.env` |
| `Не удалось залогиниться в Discord` | Неверный/сброшенный токен |
| `Не удалось зарегистрировать команду` | Неверный `CLIENT_ID`/`GUILD_ID` или бот не приглашён на сервер |
| Команда `/verify` не появляется в канале | Бот не в этом сервере; не совпадает `GUILD_ID`; не сработал `ClientReady` (см. логин выше) |
| На команду отвечают «два бота» | Запущены две сессии с одним токеном (dev + prod / два процесса) |
| Всегда «Код неверный или устарел» | Код протух (TTL), уже использован, или в БД нет записи `PENDING` с таким `verifyCode` |
